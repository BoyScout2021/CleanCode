# 의미 있는 이름

## 왜 의미 있는 이름을 지어야 하는가?

- 코드의 가독성에 가장 많은 영향을 주는 것이 이름
- 자신이 짠 클래스와 메서드도 시간이 지나면 어떤 역할을 하는지 모두 기억하지 못하기에, 이름을 통해 명확하게 역할과 작성 의도를 표현해둔다면 나중에 코드를 볼 때도 좀 더 이해가 쉬워짐

## 의미 있는 이름 짓는 방법

### 의도를 명확하게 표현하는 이름

- 변수나 함수, 클래스의 이름은 존재 이유, 수행 기능, 사용 방법을 모두 내포하고 있어야 함
  - 의도를 설명하는 데에 주석이 필요하다면, 이름에 의도를 분명히 드러내지 못했다는 것

### 혼란을 줄 수 있는 이름 피하기

- 여러가지 의미가 있는 줄임말 사용하지 않기
  - 직각삼각형의 빗변(hypotenuse)을 표현하기 위해 `hp`란 줄임말을 사용하는 경우, 코드를 읽는 사람에 따라 다르게 해석할 여지가 있음
- 특수한 의미를 지닌 단어의 잘못된 사용하지 않기
  - List 타입이 아님에도 ~~~List 라고 명명할 때, 특정 타입을 나타내는 단어가 이름에 포함되어있다면 해당 타입으로 구현되어 있다고 착각할 수 있음
- 구분이 어려운 비슷한 이름 피하기
  - `XYZControllerForEfficientHandlingOfStrings`와 `XYZControllerForEfficientStorageOfStrings`
- 일관성이 떨어지는 이름 피하기
  - 일관성이 떨어지는 이름은 IDE의 자동완성 기능을 활용하기 어렵게 함

### 의미를 명확하게 구분하기

- 단순 구분을 위한 의미 없는 숫자를 덧붙인 이름 피하기

  ```swift
  // before
  func copy<T>(t1: T, t2: inout T) {
      t2 = t1
  }
  
  // after
  func copy<T>(source: T, to destination: inout T) {
      destination = source
  }
  ```

  메서드라면, `Argument Label`을 활용해 메서드의 의도를 좀 더 명확하게 표현할 수 있음

- `a`, `an`, `the`나 `Info`, `Data`와 같은 불용어(큰 의미가 없는 단어) 사용하지 않기

  - `ProductInfo`, `ProductData` - 이름만 봐서는 어떤 차이를 가질지 예상하기 어려움

- 의미 표현이 중복되는 이름 피하기

  - 변수 이름에 `variable`을 붙이는 경우, 클래스 이름에 `class`를 붙이는 경우와 같이, 코드 선언 자체로 의미가 부여되는 경우 이름에서까지 표현해 줄 필요는 없음

### 발음하기 쉬운 이름 사용하기

- 발음하기 어려운 이름은 의사소통에도 영향을 줌
  - 여러 가지 의미가 있는 단어들을 줄여 존재하지 않는 단어를 만들어 사용하는 경우, 사람마다 발음하는 방법이 다를 수 있어 의사소통이 어려워짐

### 검색하기 쉬운 이름 사용하기

- 문자 하나만 사용하는 이름 피하기
  - `e`와 같이 문자 하나만으로 이루어진 변수를 찾으려면 `e`를 포함하고 있는 모든 이름 사이에서 찾아야 함
- 이름 길이는 범위 크기에 비례해야 함
  - 메서드 내부의 로컬 변수와 같이 한 곳에서만 사용한다면 간단한 이름도 괜찮지만, 여러 곳에서 사용한다면 검색하기 쉽도록 긴 이름을 사용해야 함

### 클래스 이름

- 클래스 이름과 객체 이름은 명사나 명사구가 적합
  - `Customer`, `WikiPage`, `Account`, `AddressParser` 등..
- `Manager`, `Processor`, `Data`, `Info`와 같이 의미 구분이 명확하지 않은 단어 사용하지 않기

### 메서드 이름

- 메서드 이름은 동사나 동사구가 적합

  - `postPayment`, `deletePage`, `save` 등..

- `Argument Label`을 활용해 메서드 호출 코드가 하나의 영어 문장처럼 읽히도록 명명하기

  ```swift
  // before
  x.insert(y, position: z)
  
  // after
  x.insert(y, at: z)
  ```

### 한 개념에 한 단어 사용하기

- 추상적인 개념 하나를 표현할 때 일관성 있게 단어를 선택하여 사용하기
  - 어딘가에서 가져온다는 개념을 표현하는 데에 `fetch`, `retrieve`, `get` 등을 섞어 사용하면, 코드를 사용할 때도 새로운 코드를 짤 때도 혼란스러움
- 추상적인 개념을 너무 크게 잡더라도 혼란을 줄 수 있으니 메서드의 맥락을 고려해 단어를 적절하게 선택해야 함
  - 추가한다는 개념을 표현하는 데에 `add`라는 단어를 사용하고 있었고, 컬렉션 타입에 값 하나를 추가하는 메서드를 작성해야 할 때 `add`를 그대로 사용하기보단, 더 적절한 의미 표현이 가능한 `insert`나 `append`를 사용해야 함

### 다른 개발자도 알 법한 단어 사용하기

- 특정 도메인에 한정된 단어를 사용하기보단, 다른 개발자들도 쉽게 이해할 수 있는 단어를 사용해야 함

### 의미 있는 맥락 추가하기

- 클래스, 함수, 네임스페이스로 맥락을 부여할 수 있음
- 클래스, 함수, 네임스페이스로도 맥락의 표현이 부족할 경우 이름에 맥락을 추가
  - 주소 일부를 표현하는 `state` 변수가 있을 때, 이 변수만 봤을 땐 주소를 의미하는지 상태를 의미하는지 알기 어려움 -> `addr`이라는 접두어를 추가하여 `addrState`라 사용하면 맥락이 좀 더 분명해짐
- 불필요한 맥락은 오히려 IDE의 자동완성 기능을 활용하기 어렵게 할 수 있으므로 주의해서 사용
  - 프로젝트 이름을 나타내기 위해 모든 클래스에 똑같은 접두사를 붙이는 경우, 모든 클래스에 같은 단어가 포함되어 있어 구분이 어려워짐
